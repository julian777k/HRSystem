## 2. 테스트 스위트별 상세 해석

### 2.1 유효성 검증 테스트 (`validation.test.ts`)

**파일 위치**: `src/__tests__/lib/validation.test.ts`
**테스트 수**: 21개 | **상태**: 전체 통과

이 테스트 스위트는 설정 마법사(Setup Wizard)에서 사용자가 입력하는 데이터의 유효성을 검증합니다.

#### 2.1.1 DB 설정 유효성 검증 (4개 테스트)

| # | 테스트명 | 검증 내용 | 의미 |
|---|---------|----------|------|
| 1 | 유효한 DB 설정은 에러가 없어야 한다 | 정상적인 DB 설정(host, port, user, password, database)을 입력하면 에러 객체가 비어있음 | 정상 입력 시 통과 확인 (Happy Path) |
| 2 | 호스트가 비어있으면 에러 | host 필드가 빈 문자열이면 `errors.host`에 에러 메시지 존재 | DB 서버 주소 필수 입력 검증 |
| 3 | 포트가 숫자가 아니면 에러 | port에 "abc" 같은 문자열 입력 시 "포트는 숫자여야 합니다" 에러 | 잘못된 포트 번호 차단 |
| 4 | DB 이름이 비어있으면 에러 | database 필드가 빈 문자열이면 에러 | 데이터베이스명 필수 입력 검증 |

**해석**: 사용자가 Setup Wizard 1단계(DB 설정)에서 잘못된 정보를 입력했을 때, 적절한 한국어 에러 메시지가 표시되어 올바른 입력을 유도합니다.

#### 2.1.2 회사 정보 유효성 검증 (3개 테스트)

| # | 테스트명 | 검증 내용 | 의미 |
|---|---------|----------|------|
| 1 | 유효한 회사 정보는 에러가 없어야 한다 | 회사명 + 올바른 URL 입력 시 에러 없음 | 정상 입력 시 통과 확인 |
| 2 | 회사명이 비어있으면 에러 | name이 빈 문자열이면 에러 | 회사명 필수 입력 검증 |
| 3 | 잘못된 URL 형식이면 에러 | "invalid-url" 등 http/https로 시작하지 않는 URL은 에러 | 서버 URL 형식 검증 |

**해석**: Setup Wizard 2단계(회사 정보)에서 회사명은 반드시 입력해야 하며, 서버 URL은 `http://` 또는 `https://`로 시작해야 합니다.

#### 2.1.3 관리자 계정 유효성 검증 (5개 테스트)

| # | 테스트명 | 검증 내용 | 의미 |

**파일 위치**: `src/__tests__/lib/setup-config.test.ts`

| # | 테스트명 | 검증 내용 | 의미 |
|---|---------|----------|------|
| 1 | 설정 파일이 없으면 null을 반환해야 한다 | 존재하지 않는 파일 경로에서 `existsSync` 결과 `false` | 최초 실행 시 설정 미완료 상태 정상 감지 |
| 2 | 설정 파일을 생성하고 읽을 수 있어야 한다 | JSON 파일 쓰기 → 읽기 → 값 확인 | 설정 저장 및 로드 정상 동작 |
| 3 | 설정 파일의 setupComplete가 false이면 미완료 상태 | `setupComplete: false` 저장 후 로드 시 `false` 확인 | 중간 저장 상태(미완료) 구분 가능 |
| 4 | 잘못된 JSON 파일은 파싱 에러를 발생시켜야 한다 | "not valid json" 문자열 저장 후 `JSON.parse` 시 예외 발생 | 손상된 설정 파일 안전 처리 |

**해석**: 시스템 설정 상태는 `data/setup-config.json` 파일로 관리됩니다. 파일이 없으면 초기 설정이 필요한 상태이고, 파일이 손상되었을 때는 예외를 발생시켜 안전하게 처리합니다.

#### 2.3.2 DB 연결 URL 생성 (3개 테스트)

| # | 테스트명 | 검증 내용 | 의미 |

| 1 | 기본 연결 URL이 올바르게 생성되어야 한다 | 일반 비밀번호 → `postgresql://msa:test123@localhost:5432/msa_hr` | 표준 PostgreSQL URL 형식 생성 |
| 2 | 특수문자가 포함된 비밀번호도 인코딩되어야 한다 | `p@ss#w0rd!` → `p%40ss%23w0rd!` (URL 인코딩) | 특수문자 비밀번호 안전 처리 |

## 6. 품질 평가 요약

### 6.1 강점

- **근로기준법 준수**: 연차 계산 로직이 대한민국 근로기준법 제60조를 정확히 반영
- **한국어 에러 메시지**: 실제 사용자가 이해할 수 있는 한국어 검증 메시지
- **경계값 테스트**: 연차 계산에서 최대값(25일), 최소값(0일), 전환점(3년차) 등 경계값을 철저히 테스트
- **보안 검증**: 비밀번호 최소 길이, 이메일 형식, URL 인코딩 등 보안 관련 검증 포함
- **빠른 실행**: 0.3초 이내 전체 테스트 완료 (개발 생산성 유지)

### 6.2 개선 필요 사항

| 항목 | 현재 상태 | 개선 방향 |
|------|----------|----------|
| 소스 파일 import | 테스트 내 함수 직접 정의 | 소스 파일에서 함수 분리 후 import하여 커버리지 수집 |
| API 통합 테스트 | 수동 curl 테스트만 수행 | supertest 또는 Next.js testClient로 자동화 |
| UI 컴포넌트 테스트 | 미작성 | React Testing Library로 Setup Wizard 렌더링 테스트 추가 |
| DB 연동 테스트 | 미작성 | 테스트 전용 DB 컨테이너로 Prisma 쿼리 검증 |
